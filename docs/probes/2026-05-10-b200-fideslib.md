This kernel uses encrypted per-token inputs and encrypted recurrent state, with
public plaintext weights. The immediate bottleneck is not GPU throughput but
the naive level alignment strategy: 28 of 44 ciphertext-plaintext multiplies in
the default run are unity multiplies inserted only to align fresh encrypted
inputs with the recurrent state level. The next implementation step should make
level-aware input/update encoding explicit instead of paying this alignment
tax.

The current default moves the public-weight multiply to the client side under
the baseline threat model (`user data private, model public`). The client
encrypts the already weighted update `B_t x_t` at the target CKKS level. This
keeps the model public and the user input encrypted, while removing the
server-side alignment tax.

Offset sweep at `seq_len=8`:

| Job | Input mode | Offset | ct-pt mul | Unity alignment mul | Latency/token |
| --- | --- | ---: | ---: | ---: | ---: |
| 9991 | client-update | 0 | 8 | 0 | 0.000771 s |
| 9992 | client-update | 1 | 9 | 1 | 0.000840 s |
| 9993 | client-update | 2 | 10 | 2 | 0.009017 s |
| 9994 | server-bx | 0 | 44 | 28 | 0.000664 s |

The `seq_len=8` wall-clock numbers are too small to treat as stable, but the
