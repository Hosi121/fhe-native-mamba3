
This kernel uses encrypted per-token inputs and encrypted recurrent state, with
public plaintext weights. The immediate bottleneck is not GPU throughput but
the naive level alignment strategy: 28 of 44 ciphertext-plaintext multiplies in
the default run are unity multiplies inserted only to align fresh encrypted
inputs with the recurrent state level. The next implementation step should make
level-aware input/update encoding explicit instead of paying this alignment
tax.

The current default moves the public-weight multiply to the client side under
the baseline threat model (`user data private, model public`). The client
encrypts the already weighted update `B_t x_t` at the target CKKS level. This
keeps the model public and the user input encrypted, while removing the
server-side alignment tax.

path. This is the measurement target for SSD scan and more aggressive packing.

## Native MIMO Rank Sweep

Added `slurm/fideslib_stage0_sweep.sbatch` so a single SLURM allocation builds
the native kernel once and runs multiple configurations. The sweep records
failed configurations as JSON and continues, avoiding a lost allocation when one
packing group is invalid.

`seq_len=8`, `d_state=4`, `readout_mode=rank-reduce`, `ring_dim=4096`:

| --- | ---: | ---: | --- | ---: | ---: | ---: | ---: | --- |
| 10006 | 1 | 4 | ok | 40 | 16 | 0.009985 | 0.001248 | output error 1.08e-15 |
| 10006 | 2 | 8 | ok | 48 | 24 | 0.012840 | 0.001605 | output error 6.94e-16 |
| 10006 | 4 | 16 | failed | - | - | - | - | CKKS decode error |
| 10006 | 8 | 32 | failed | - | - | - | - | CKKS decode error |

Increasing `ring_dim` to 8192 did not fix rank 4/8 readout in job 10007.
The recurrence-only control in job 10008 succeeded for rank 4 and rank 8, so
the failure is isolated to the rank-reduce readout/rotation path rather than
the recurrent state update itself.

Short-sequence debugging showed rank 4 readout can decrypt to nonfinite output
even at `seq_len=1`. Job 10010 now emits valid JSON for this case:

```json
{
  "model": {"seq_len": 1, "d_state": 4, "mimo_rank": 4},
  "output_has_nonfinite": true,
  "output_max_abs_error": 1e+300,
  "decrypted_final_output": [null, null, null, null],
  "readout_rotation_keys": [1, 2, 3, 6, 9]
}
```

Current interpretation: packing multiple rank groups into one ciphertext works
for recurrence, but the current rank-reduce readout layout is only verified up
to `mimo_rank=2` under the toy CKKS parameters. The next implementation step is
to replace this scatter-heavy readout with a layout that avoids large intra-ct
rotations for higher packing groups, or to split readout groups before trying
larger MIMO/head packs.
